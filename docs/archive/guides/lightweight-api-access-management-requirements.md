# 轻量 API 接入管理机制需求（API Center，非重型网关）

## 1. 背景

当前系统对外部 OpenAPI endpoint 的接入方式偏“导入即发布”，缺乏统一的接入治理与运行期状态管理，主要问题：

- 缺少 endpoint 分类、生命周期和上线/下线机制
- 缺少主动可用性检测与预警，常在 Tool 调用失败后才暴露问题
- 缺少发布前筛选与准入控制，无法做分阶段发布

需要建设一个“轻量 API Center”能力，重点做接入管理和状态治理，不引入重型 API 网关转发链路。

## 2. 目标与边界

### 2.1 目标

- 统一管理自动导入与手工注册 endpoint
- 建立 endpoint 分类、状态、发布、下线的全生命周期能力
- 引入主动健康检测与预警，保障 Tool 发布质量

### 2.2 边界（明确不做）

- 不承担全流量代理、七层转发、复杂限流熔断编排
- 不替代现有业务网关
- 不做多集群复杂服务网格

## 3. 核心能力要求

### 3.1 API 接入中心（统一入口）

支持两类接入源：

- 自动导入：OpenAPI URL/文件
- 手工注册：补录 endpoint（method/path/参数/认证方式）

统一进入 `endpoint_registry`，形成标准记录。

### 3.2 Endpoint 分类管理

支持以下分类维度：

- 业务域（如 user/pet/order）
- 来源系统（第三方/内部）
- 认证类型（none/apiKey/bearer/oauth）
- 风险等级（low/medium/high）

分类用于筛选、发布策略、巡检策略分层。

### 3.3 生命周期与状态管理

建议状态：

- `draft`：已接入待审核
- `verified`：基础校验通过
- `published`：允许发布为 Tool
- `degraded`：可用性下降，受限发布
- `offline`：下线，不可发布
- `retired`：废弃

状态由人工动作 + 自动检测共同驱动。

### 3.4 发布与下线控制

支持 endpoint 到 Tool 的“选择性发布”，而非全量发布：

- 按分类、来源、状态、标签筛选发布
- 支持灰度发布（小范围模型/租户）
- 一键下线 endpoint 对应 tools

### 3.5 主动检测与预警

检测机制（轻量）：

- 定时健康检查（HEAD/GET，或轻量探测请求）
- 认证有效性检查（token/密钥失效检测）
- schema 变更侦测（与历史版本对比）

预警机制：

- 连续失败阈值触发告警
- 从 `published` 自动降级到 `degraded/offline`（可配置）
- 在管理界面展示“不可发布原因”

## 4. 手工注册 endpoint 管理要求

- 手工录入模板：`method/path/baseUrl/认证/参数/schema`
- 支持“最小必填 + 后续补全”模式
- 支持与自动导入 endpoint 合并去重（同源同路径同方法）
- 支持标记“手工维护优先”，避免被自动导入覆盖

## 5. 数据模型建议（SQLite）

建议新增/扩展：

- `endpoint_registry`：接入主表（来源、分类、状态、认证、版本）
- `endpoint_health_log`：探测结果与错误上下文
- `endpoint_lifecycle_event`：状态变更审计
- `tool_publish_record`：发布批次与 endpoint 清单
- `alert_event`：告警记录与恢复记录

## 6. 管理端需求（UI）

- 接入页：导入 OpenAPI、手工注册
- 列表页：分类筛选、状态筛选、批量动作
- 详情页：健康趋势、最近错误、发布历史、状态流转
- 发布页：选择 endpoint -> 预检查 -> 发布/灰度/下线

## 7. 与语义层协同关系

发布 Tool 的前置条件建议为：

1. endpoint 状态 `published` 且健康检查通过
2. 语义层状态为 `published`
3. 权限与可见性策略通过

任一条件不满足，不允许对外发布。

## 8. 实施步骤（4 阶段）

### 阶段 1：接入台账化

- 建立 `endpoint_registry`
- 打通自动导入 + 手工注册统一入库
- 列表查询与状态基础流转

### 阶段 2：健康检测与告警

- 定时探测任务（可从 cron/job 启动）
- 健康日志与告警记录
- 失败阈值触发状态降级

### 阶段 3：发布管控

- 筛选发布、批量发布、下线动作
- 发布前检查（状态/语义/认证）
- 发布记录审计

### 阶段 4：运营闭环

- 健康看板与趋势
- 按来源/分类故障分析
- 恢复确认与复盘流程

## 9. 验收标准

- 所有 endpoint 都有可追踪状态与来源
- 新增故障可在阈值窗口内被主动发现并告警
- 非 `published` endpoint 不得发布为 Tool
- 支持手工注册 endpoint 的完整创建、审核、发布、下线流程

## 10. 价值与必要性结论

- 有必要建设“轻量 API Center”，因为它解决的是“接入治理和发布管控”问题，而非流量网关问题。
- 当前阶段不建议直接引入重型 API 网关；先把 endpoint 生命周期、健康检测、语义发布、权限策略做完整，性价比最高、落地最快。

## 11. 平台与数据库兼容性要求（补充）

本需求为硬性约束：

1. 数据库必须同时支持 `SQLite` 与 `PostgreSQL`。
2. 部署与运行必须同时支持 `Linux` 与 `Windows`。
3. 接入管理、状态管理、健康检测、告警、发布/下线能力需在双数据库与双平台保持一致行为。
4. 验收时必须提供四象限验证结果（Linux/Windows x SQLite/PostgreSQL）。
