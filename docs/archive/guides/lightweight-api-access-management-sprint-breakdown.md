# 轻量 API 接入管理 Sprint 拆解（非重型网关）

## 1. 目标

建设“轻量 API Center”能力，统一管理自动导入与手工注册 endpoint，并形成可发布、可下线、可预警的治理闭环。

## 2. Sprint 建议（4 个 Sprint）

### Sprint 1：统一接入台账

后端任务：

1. 扩展 `endpoint_registry`：
   - 来源（自动导入/手工注册）
   - 分类（业务域、认证类型、风险等级）
   - 生命周期状态（`draft/verified/published/degraded/offline/retired`）
2. 新增手工注册 API（最小必填）
3. 增加去重逻辑（同源同 path 同 method）
4. 增加“手工优先”标记，防自动导入覆盖

前端任务：

1. 新增“手工注册 endpoint”表单
2. endpoint 列表页增加分类与状态筛选

测试任务：

1. 手工注册用例测试
2. 去重与覆盖策略测试

交付标准：

1. 自动导入和手工注册都进入统一台账
2. 可通过状态和分类快速筛选 endpoint

### Sprint 2：健康检测与状态驱动

后端任务：

1. 新增 `endpoint_health_log`
2. 增加定时探测任务（HEAD/GET 轻探测）
3. 增加认证有效性探测（凭证失效识别）
4. 连续失败阈值驱动状态降级（`published -> degraded/offline`）
5. 新增 `endpoint_lifecycle_event` 记录状态变更审计

前端任务：

1. endpoint 详情页展示健康趋势和最近错误
2. 显示“不可发布原因”

测试任务：

1. 健康探测成功/失败测试
2. 阈值降级逻辑测试
3. 状态审计记录测试

交付标准：

1. endpoint 可用性可被主动发现，不依赖 Tool 失败后暴露
2. 状态可被自动检测机制驱动

### Sprint 3：发布管控与下线闭环

后端任务：

1. 新增 `tool_publish_record` 记录发布批次
2. 发布前检查：
   - endpoint 状态必须 `published`
   - 健康探测通过
   - 语义状态为 `published`
3. 增加批量发布/批量下线 API
4. 增加灰度发布能力（按标签或租户范围）

前端任务：

1. 发布页：筛选 -> 预检查 -> 发布/灰度/下线
2. 发布记录查询页

测试任务：

1. 发布准入拦截测试
2. 批量发布回归测试
3. 下线后 Tool 不可见测试

交付标准：

1. 实现“选择性发布”而非全量发布
2. 非合规 endpoint 无法被发布

### Sprint 4：预警与运营闭环

后端任务：

1. 新增 `alert_event`（告警和恢复）
2. 告警规则：
   - 连续失败次数阈值
   - 关键 endpoint 单次失败立即告警（可配置）
3. 恢复检测与自动关闭告警
4. 增加故障聚合视图 API（按来源/分类）

前端任务：

1. 告警中心页（当前告警、历史告警、恢复状态）
2. 运营看板（可用率、降级率、离线数量）

测试任务：

1. 告警触发与恢复测试
2. 运营看板数据一致性测试

交付标准：

1. 故障告警可自动触发与恢复闭环
2. 管理端具备日常运营可视化能力

## 3. 工作分解（按角色）

后端：

1. 台账建模与状态机
2. 健康探测任务与告警逻辑
3. 发布管道准入控制

前端：

1. 接入管理页面
2. 健康与告警可视化
3. 发布与下线操作页

测试：

1. 状态机、阈值、准入策略单元测试
2. 导入/手工注册/发布/下线集成测试
3. 运营数据端到端一致性测试

运维：

1. 定时任务运行监控
2. 告警通知通道接入（邮件/IM）
3. 指标仪表板落地

## 4. 依赖与风险

关键依赖：

1. 与语义层发布状态联动
2. endpoint 认证配置可被安全读取
3. 定时任务稳定运行

主要风险与缓解：

1. 探测频率过高影响外部服务  
   缓解：分级频率、指数退避、白名单豁免
2. 误报告警过多影响运营  
   缓解：连续失败阈值与抖动窗口
3. 手工与自动配置冲突  
   缓解：引入“手工优先”与冲突审计

## 5. 里程碑与验收

里程碑 A1：

1. 接入台账与手工注册上线（Sprint 1）

里程碑 A2：

1. 主动检测与状态降级上线（Sprint 2）

里程碑 A3：

1. 发布准入与批量下线上线（Sprint 3）

里程碑 A4：

1. 告警中心与运营闭环上线（Sprint 4）

总体验收口径：

1. 100% endpoint 可追踪来源、状态、最近健康状态
2. 关键 endpoint 故障可在约定窗口内告警
3. 非发布态 endpoint 不能进入最终 Tool 集合

## 6. 平台与数据库兼容性基线（新增硬性约束）

必须同时满足以下要求：

1. 数据库：`SQLite` 与 `PostgreSQL` 双支持，功能行为一致。
2. 操作系统：`Linux` 与 `Windows` 双支持，运行与测试流程一致。

实施要求：

1. 接入台账、健康日志、告警与发布记录等表结构需跨数据库兼容。
2. 健康探测与调度实现不能依赖单系统命令或路径规则。
3. 发布与下线流程中的文件路径、日志编码、脚本调用保持跨平台一致。
4. CI 至少包含四组矩阵验证：
   - Linux + SQLite
   - Linux + PostgreSQL
   - Windows + SQLite
   - Windows + PostgreSQL

补充验收口径：

1. 两种数据库下 endpoint 全生命周期流程一致可用。
2. Linux/Windows 下健康检测、告警、发布与下线流程一致可用。
